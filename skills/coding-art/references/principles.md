# 编程工艺 66 条核心原则

> 提炼自《代码大全 2》(Steve McConnell) 与《UNIX 编程艺术》(Eric S. Raymond)，融合软件工程最佳实践。

---

## 一、心智与哲学（8 条）

> 编程的根本态度与价值观。这些原则决定了在具体技术选择之前，工程师如何看待问题。

### #1 管理复杂度是软件的首要技术使命

任何设计决策都应问："这降低了还是增加了复杂度？" 复杂度分**本质复杂度**（问题固有）与**偶然复杂度**（方案引入）。设计应消灭方案引入的复杂度、驯服问题固有的复杂度。

- **反面信号**：一个改动牵涉多处、新人需要数周才能理解某个模块、修改一处引发意外故障
- **反面信号**：一个新功能引入了太多新的依赖，或者对原有设计造成了过多影响

### #2 数据压倒一切——核心是数据结构，而非算法

如果已经选择了正确的数据结构并把一切组织得井井有条，正确的算法也就不言自明。先想清楚"数据是什么形态"，再想"如何处理它"。

- **反面信号**：用复杂算法弥补错误的数据结构选择、到处用 List 而忽略 Map/Set 的适用性
- **正面信号**：用正确的数据结构，结合该结构本身就支持的功能特性，解决了之前用代码实现的功能

### #3 策略与机制分离

机制（底层操作、核心逻辑）应稳定长久；策略（界面观感、业务规则、配置项）应可灵活替换。把策略推到上层，机制沉到下层，系统才能持久保鲜。

- **反面信号**：核心逻辑里硬编码 UI 文案、业务规则散落在底层工具函数中
- **正面信号**：核心逻辑始终保持结构清晰，不被不相关的业务逻辑所污染，策略层/易变的逻辑合理的抽象为配置或实现层代码

### #4 尽可能简单，但别太简单

简单性是控制复杂度的核心手段。但过度简化会丢失必要的功能、灵活性或安全性。目标是"恰到好处的简单"——能满足真实需求的最简方案。

- **反面信号**：为了"简洁"省掉了必要的错误处理、为了"优雅"引入了过度抽象

### #5 隔离变化点

找出系统中可能变化的部分，用胶合层或接口把它们与稳定部分隔离开。变化被局限在局部，不会扩散影响全局。隔离变化 = 控制风险。

- **反面信号**：一个外部 API 变动需要修改十几个文件、业务规则变更牵连核心逻辑

### #6 一个程序/模块只做一件事，并做好

功能单一、职责清晰。新任务应通过新模块解决，而非往旧模块里堆砌功能。这是 Unix 哲学的第一条，也是单一职责原则的源头。

- **反面信号**：模块名出现"和/And"、一个类有 20 个公开方法且语义发散

### #7 透明性优于聪明

好的系统应该是透明的——通过观察输入输出和中间状态，就能推断其内部行为。不透明的"聪明"代码是调试和运维的噩梦。

- **反面信号**：必须读源码才能理解行为、行为依赖隐式全局状态、调试需要"魔法步骤"
- **正面信号**：命名清晰易懂，抽象层次合理，代码结构清晰，函数化程度高，易于编写测试

### #8 先正确，再高效

不要为性能牺牲正确性与可读性。大多数代码的性能瓶颈只出现在少数热点。先把代码写对、写清楚，再在测量指导下优化真正的瓶颈。

- **反面信号**：为省一次函数调用内联了本该独立的方法、用位运算替代除法导致可读性骤降
- **正面信号**：先将可维护性放在第一位，如果对性能拿不准的地方，可以通过询问人类来确定

## 二、设计与模块化（10 条）

> 架构、模块、抽象、耦合。设计的好坏决定了后续编码的难易。

### #9 封装是模块设计的第一原则

模块应隐藏内部实现细节，只暴露稳定的接口。封装使模块可以被独立理解、测试和替换。每个模块都应有自己负责的"秘密"。

- **反面信号**：外部代码直接访问模块内部字段、内部数据结构泄漏到接口签名中

### #10 保持模块间的松散耦合

耦合越松，模块越可独立理解、测试、复用。耦合度由规模、可见性、灵活性共同决定，应优先小、可见、灵活的耦合。

- **反面信号**：改 A 模块必须同步改 B 模块、模块间通过共享全局变量通信

### #11 追求强内聚

模块内的元素应紧密相关、共同实现一个明确目标。内聚强度排序：功能内聚（最强）> 顺序内聚 > 通信内聚 > 时间内聚 > 逻辑内聚 > 偶然内聚（最弱）。

- **反面信号**：模块内函数彼此无关、命名困难（找不到一个词概括模块职责）

### #12 遵守 SPOT 原则——Single Point of Truth

任何一个知识点在系统中都应当只有单一、明确、权威的表述。重复的知识点是维护的噩梦——改了一处忘了改另一处，数据就不一致了。

- **反面信号**：同一配置值出现在三处、业务规则在前后端各写一遍、常量散落多个文件

### #13 保持正交性

正交性意味着改变 A 不会影响 B。系统中的各功能维度应相互独立，避免一个旋钮同时控制多个不相关的特性。

- **反面信号**：调整日志格式影响了业务逻辑、修改缓存策略改变了接口行为

### #14 用层次化结构组织系统

复杂的系统应分解为层次结构，每层只依赖下层提供的抽象。跨层调用会破坏层次，增加理解成本与耦合。

- **反面信号**：业务层直接访问数据库驱动、UI 代码里写 SQL、底层工具反向调用高层服务

### #15 接口应一致、完整、最小

接口要一致（同类操作签名风格统一）、完整（覆盖必要操作）、最小（不提供不需要的功能）。一个接口不应同时展现多种互相矛盾的抽象。

- **反面信号**：同类操作有的用 `getX` 有的用 `fetchX`、接口暴露了 50 个方法但只用 5 个

### #16 组合（包含）往往优于继承

组合在运行时可替换、耦合更松、不引入父类包袱。能用组合解决的，优先用组合。只有严格满足 "is-a" 关系时才用继承。

- **反面信号**：为复用少量代码而继承、继承层次超过 3 层、子类覆盖父类方法却改变语义

### #17 子类必须能替换父类（Liskov 替换原则）

子类不应削弱父类的契约。若某处使用父类，换成子类后行为仍应正确，否则继承设计有误。

- **反面信号**：子类方法抛出父类不会抛的异常、子类对某些输入返回与父类语义不同的结果

### #18 胶合层应尽可能薄

胶合层（如 API 适配层、转换脚本）的任务是连接而非重新实现。越薄的胶合层，维护负担越小、性能损失越少。

- **反面信号**：适配层里重新实现了被适配方的逻辑、胶合层比两端代码加起来还长

## 三、命名与表达（8 条）

> 代码可读性的第一道关。好名字胜过注释。

### #19 名字应描述"是什么"而非"怎么做"

名字应反映问题域语义（如 `employeeCount`、`pendingOrders`），而非实现细节（如 `i`、`dataArray`、`ptr`）。

- **反面信号**：变量名暴露数据结构（如 `hashMap` 而非 `userIndex`）、用类型缩写命名

### #20 名字长度应与作用域匹配

作用域小的变量（如循环计数）可短；作用域大的变量应长且具描述性，让远处使用也能自解释。全局变量必须长名。

- **反面信号**：全局变量叫 `g`、循环里用 `employeeIndex` 而非 `i`

### #21 遵循统一的命名约定

项目内约定（大小写、前缀、缩写规则）应一致执行。**约定的存在比具体形式更重要**——团队统一用 camelCase 还是 snake_case 不重要，但必须统一。

- **反面信号**：同一项目里 `getUserInfo`、`get_user_info`、`userinfo` 并存

### #22 避免误导性与含糊的名字

避免用 `accountList` 表示非 List 类型；避免 `data`、`info`、`temp`、`value` 这类无信息量名字。误导性名字比没有名字更糟。

- **反面信号**：`accountList` 实际是 Set、`flag` 既非真也非假而是状态码

### #23 布尔变量用肯定式命名

用 `isDone` 而非 `isNotDone`。否定式命名会让 `if (!isNotDone)` 这类判断难以理解。布尔变量应回答"是/否"问题。

- **反面信号**：`if (!isNotEnabled)`、`isDisabled == false`

### #24 用命名常量替代魔法数字

代码中的字面量数字（如 `86400`、`0.05`）应替换为有名字的常量（如 `SECONDS_PER_DAY`、`TAX_RATE`）。名字承载含义，便于搜索与修改。

- **反面信号**：代码里出现 `if (status == 3)`、多处重复同一个未命名的数字

### #25 代码应自说明

好的命名、清晰的结构、合理的模块划分能让代码自解释。注释只补充代码无法表达的内容（意图、权衡、约束、参考）。

- **反面信号**：变量叫 `data` 靠注释解释它是什么、注释重复代码已有的信息

### #26 注释应解释"为什么"，而非"是什么"

注释重复代码已有的信息是噪音；有价值的是解释**意图**（为什么这样做）、**权衡**（为什么不那样做）、**约束**（外部限制）、**参考来源**（算法出处、规范链接）。

- **反面信号**：`i++; // i 加 1`、注释描述代码行为而非设计理由

## 四、函数设计（8 条）

> 子程序（函数/方法）是代码的基本构件。高质量函数让人一眼读懂意图。

### #27 每个函数只做一件事

函数应具有强功能内聚：只完成一个明确定义的任务。若名称里出现"和/And"，多半职责过多。一个函数应在抽象层次上保持一致。

- **反面信号**：函数名 `loadAndParseAndValidate`、函数同时做 IO 和纯计算且无法分开测试

### #28 给函数起强描述性的名字

好名字胜过注释。动词+宾语结构描述行为（`calculateTotal`、`validateInput`），避免含糊词（如 `process`、`handle`、`doStuff`）。**名字过长优于名字含糊**。

- **反面信号**：`process()`、`handleData()`、`doIt()`

### #29 控制函数的长度

函数过长通常意味着职责过重或缺少抽象。但也不应为了短而拆分到逻辑支离破碎——**可读性是最终判据**，不是行数。

- **反面信号**：单函数超过 100 行、嵌套超过 4 层、需要滚动多屏才能看完

### #30 尽量减少参数数量

参数越多，调用越易出错、测试越难（组合爆炸）。超过 7 个参数通常意味着该把部分数据聚合为对象。零参数最好，一二个次之。

- **反面信号**：`createUser(name, age, email, phone, address, role, dept, manager, startDate)`

### #31 避免使用输出参数

输出参数让调用点难以理解（看不出函数会修改传入对象）。优先用返回值传递结果；确需多值返回时考虑返回对象或重构职责。

- **反面信号**：`void process(Order order)` 实际会修改 order、调用者看不出 `transform(data)` 会改 data

### #32 区分函数与过程

有返回值（函数）用于"求值"，无返回值（过程）用于"产生副作用"。**不要让函数既求值又改状态**——这会让调用者无法预期行为。

- **反面信号**：`getUser(id)` 同时修改了用户最后登录时间、`calculate` 函数有副作用

### #33 函数应在接口处说明其契约

前置条件、后置条件、对参数的要求、返回值含义、异常情况，都应在接口或注释中讲清楚。契约是调用者与被调用者的协议。

- **反面信号**：调用者不知道传 null 会怎样、返回值 -1 是错误码还是合法值

### #34 让循环退出路径清晰可见

循环应让读者容易判断何时继续、何时结束。多个 `break` / `return` / `continue` 交织会增加理解成本；但清晰的守卫语句、短函数中的 early return、或语义明确的 `continue` 往往比强行单一出口更可读。目标不是机械单出口，而是退出路径清晰、局部、可验证。

- **反面信号**：循环里有多个相互交织的 `break`、`continue` 和 `return`，退出条件分散且缺少命名

## 五、控制流与数据（6 条）

> 语句级别的正确性。这是写出无 bug 代码的基本功。

### #35 简化布尔表达式

复杂布尔表达式易错且难读。用德摩根律、引入中间布尔变量、提前返回等方式简化。布尔表达式应能被大声朗读出来仍通顺。

- **反面信号**：`if ((a && !b) || (c && d) || (!a && e && !f))`、嵌套 3 层以上条件

### #36 把复杂条件分解为命名的布尔变量

将 `if (a && !b || c)` 拆为 `isReady = a && !b; if (isReady || c)`。命名的布尔变量让条件含义自解释，也便于复用。

- **反面信号**：条件表达式超过一行、同一条件在多处重复出现

### #37 条件表达式尽量用正向表达

正向条件（"如果满足 X 则做 Y"）比负向条件（"如果不满足 X 则做 Y"）更易理解。`if (isValid)` 优于 `if (!isInvalid)`。

- **反面信号**：`if (!isNotEnabled)`、`unless (!flag)`

### #38 把最常见的分支放在 if 而非 else

按"正常路径在前、异常路径在后"组织，让读者顺着主流程阅读。错误处理用守卫语句提前返回，避免深层嵌套。

- **反面信号**：主逻辑被 `if (error) {...} else { 主逻辑 }` 包裹在 else 里

### #39 用表驱动法替代复杂的 if-else / switch

当逻辑分支由数据决定时，用查表（数组、字典）替代长串条件判断，让数据与逻辑分离，便于扩展。

- **反面信号**：switch 有 20 个 case、每加一个状态都要改 if-else 链

### #40 警惕数值的溢出、精度与边界

整数运算注意溢出；浮点数注意精度丢失（不要用 `==` 比较浮点数，用阈值）；数组/容器访问前检查边界。对临界值做边界测试。

- **反面信号**：`if (a + b > INT_MAX)` 写法错误、`if (0.1 + 0.2 == 0.3)` 返回 false

## 六、防御式编程（6 条）

> 不要相信任何输入与假设。让代码在被错误使用时仍能体面地失败。

### #41 用断言记录"不应发生"的假设

断言用于检测编程错误（永远不应为真的条件），在生产中可关闭。**断言不应用于处理预期错误**——预期错误要用正常的错误处理路径。

- **反面信号**：用断言检查用户输入、断言失败导致生产环境崩溃（除非在交易用户输入场景且有良好的错误捕获机制）

### #42 校验所有外部输入

对来自用户、文件、网络的输入一律存疑，校验后再使用。"垃圾输入"不应导致崩溃或数据损坏。明确信任边界，边界外必验，边界内可放宽。

- **反面信号**：直接 `JSON.parse` 未校验的请求体、未检查数组长度就访问索引

### #43 制定统一的异常处理策略

明确哪些错误用异常、哪些用返回码；异常应携带足够上下文（发生了什么、在哪里、为什么）；**不要捕获后吞掉**（空 catch）。

- **反面信号**：`catch (Exception e) {}`、异常无堆栈信息、同一项目混用 3 种错误处理风格

### #44 错误信息应精确且可操作

错误信息不应只说"出错了"或"失败"，而应说明：**什么操作失败了、为什么失败、用户能做什么来修复**。错误信息是给人和工具看的。

- **反面信号**：`throw new Error("error")`、`return -1`、错误信息只有错误码无说明

### #45 在"尽早失败"与"优雅降级"间取得平衡

对**编程错误**尽早暴露（fail fast）——越早发现越易定位；对**外部错误**尽可能优雅降级——用户体验优先。区分这两类错误是关键。

- **反面信号**：空指针一路传播到 UI 才报错、网络超时让整个服务崩溃而非降级

### #46 提供 dry-run（试运行）模式

destructive 操作（如删除、批量修改、迁移）应提供试运行模式，让用户在真正执行前预览将要发生什么。可逆操作优先于不可逆操作。

- **反面信号**：删除命令无 `--dry-run`、迁移脚本无预览直接改库

## 七、接口与协议（6 条）

> 对外契约与协作。好的接口让协作自然发生。

### #47 面向人和工具链的接口优先使用文本

面向人、脚本、配置、日志、CLI 和长期调试的接口，优先使用文本格式（如 JSON、YAML、CSV、INI）。文本接口可被 grep、可被人工阅读、可被未来未知的程序消费。对高吞吐、强 schema、低延迟、低带宽或列式分析等场景，二进制格式（如 Protobuf、Arrow、Parquet、MessagePack）可能更合适。关键是让接口可理解、可演进、可调试，而不是机械排斥二进制。

- **反面信号**：配置、日志、CLI 输出使用专有二进制格式；在没有性能或 schema 证据时自定义难调试协议

### #48 数据格式应尽可能简单

简单的数据格式更容易被人类阅读和工具处理。不要为了节省几个字节而引入复杂编码。遵循已有约定（JSON、CSV、INI、Protobuf 等），不重新发明格式。

- **反面信号**：自定义分隔符、用复杂嵌套结构表达本可扁平化的数据

### #49 为协议设计清晰的边界与终止机制

无论是空行分隔、长度前缀、还是特殊终止符，协议必须有明确的边界定义，避免解析歧义。元数据应内嵌于数据流中，而非依赖隐式约定。

- **反面信号**：靠"读到这里就结束"的模糊边界、依赖连接关闭判断消息结束

### #50 接口设计遵循最小惊喜原则

用户输入 X，程序输出 Y，这种对应关系应尽可能直观。违反用户直觉的行为（即使"技术上正确"）是糟糕的设计。一致性是降低学习成本的关键。

- **反面信号**：`getDate()` 返回时间戳而非日期对象、`setX(null)` 会删除 X 而非忽略

### #51 提供合理的默认值

用户不应为每个选项都做出选择。软件应在不提供配置时也能以合理方式运行。**默认值体现了设计者的品味**——好的默认值让 80% 的用户无需配置。

- **反面信号**：必须配置 10 个参数才能运行、默认值导致不安全的行为（如默认关闭加密）

### #52 用户可见输出保持安静，但错误和观测信号必须清楚

对 CLI 和脚本来说，成功路径应尽量只输出真正需要的信息，便于管道组合；但错误必须明确报告，不能悄无声息地失败。对服务端、后台任务和 Agent，"安静"不等于缺少可观测性：用户可见输出可以克制，日志、metrics、trace 和审计事件必须足够支持排障。

- **反面信号**：CLI 成功时打印一堆噪音；失败时静默返回默认值；服务端缺少必要日志、指标或 trace

## 八、并发与状态（4 条）

> 状态管理是软件正确性的难点，并发是 bug 的温床。

### #53 避免共享可变状态

共享可变状态是并发 bug 的主要来源。尽量通过消息传递而非共享内存来通信；如果必须共享，用锁保护好边界。**不可变对象天生线程安全**。

- **反面信号**：全局可变单例、多线程直接读写同一 HashMap、无同步的 lazy init

### #54 最小化状态，显式化状态

能用无状态函数就不用有状态对象；能用局部变量就不用成员变量；能用私有就不用公开。隐藏在全局变量、环境变量、临时文件中的状态会让系统行为不可预测。

- **反面信号**：用全局变量传参、用静态字段缓存中间结果、依赖环境变量的隐式配置

### #55 让并发模型与问题匹配

不要一刀切地使用线程或进程。I/O 密集型任务适合事件驱动或异步；CPU 密集型任务适合多进程/多线程。选择最自然、最简单的模型。远程通信与本地通信尽量使用相同的抽象。

- **反面信号**：为 I/O 密集任务开线程池阻塞等待、用多进程做本可异步的 HTTP 请求

### #56 用分离的单元实现分离的 concern

当两个任务在逻辑上独立时，用独立的进程/线程/协程实现它们，而不是在同一个执行流里纠缠。分离的单元故障隔离更好、扩展性更强。

- **反面信号**：一个服务同时处理计算和 IO 且互相阻塞、后台任务和前台请求共享线程池

## 九、测试与质量（6 条）

> 质量不是事后检查出来的，而是构建过程中持续注入的。

### #57 质量是构建期的目标，而非事后检查

质量应在构建阶段就追求，而非"先写完再修"。缺陷预防的成本远低于修复（需求期修复成本约为维护期的 1/100）。高质量的构建直接降低下游维护成本。

- **反面信号**："先写完再说，后面再补测试"、把质量全推给 QA 团队

### #58 代码检查发现缺陷的能力强于测试

代码检查（review/inspection）能发现测试难以覆盖的缺陷类型（如遗漏、错误类型、设计缺陷），应作为常规质量手段。结对编程是即时检查的轻量形式。

- **反面信号**：只有 CI 跑测试没有 PR review、review 只看风格不看逻辑

### #59 测试先行能改善设计与质量

先写测试再写代码（或同步进行），会倒逼代码可测试、接口清晰，同时提供即时反馈。可测试性差的代码通常设计也有问题。

- **反面信号**：写完才补测试且测试极难写、私有方法无法测试说明设计耦合

### #60 测试应覆盖边界与等价类

用等价类划分减少冗余用例，用边界值分析覆盖易错边界。**不要只测"正常路径"**。null、空集合、最大值、负数、并发访问都是边界。

- **反面信号**：只测 happy path、边界值从未测试、并发场景无测试
- **出处**：[CC] 第 22 章

### #61 用科学方法调试，而非盲目改代码

复现缺陷 → 形成假设 → 设计实验验证假设 → 修复 → 回归测试。盲目"改改试试"会让代码更糟，且即使"修好了"也不知道为什么。每次只改一个变量。

- **反面信号**：随机加 `if` 试图消除报错、改到不报错就提交、修复后无回归测试

### #62 重构应小步进行并保持测试通过

每次只做一种小改动（重命名、提取方法等），改完即跑测试，确保行为不变。大步重构易引入缺陷且难以定位。数据级重构与语句级重构并重。

- **反面信号**：一次重构改 500 行、重构期间测试红了一周、重构与功能修改混在一个提交

## 十、优化与工程（4 条）

> 性能、工具、文档与工艺修炼。这是工程师长期成长的维度。

### #63 优化前必须先测量

凭直觉判断热点常常出错。用 profiler 或计时工具测量后，针对真正的瓶颈优化，而非猜测。性能遵循 80/20 法则——约 80% 的执行时间花在 20% 的代码上。

- **反面信号**："我觉得这里慢"就改、优化后不测量效果、优化了非热点代码

### #64 优先优化算法与数据结构，再做代码级微调

把 O(n²) 换成 O(n log n) 的收益，远大于在 O(n²) 代码里抠几条指令。选对数据结构，算法自然清晰。避免过早优化——"过早优化是万恶之源"（Knuth）。

- **反面信号**：在 O(n²) 算法上手写汇编、用位运算省一个时钟周期却保持 O(n²)

### #65 重用而非重新发明

如果已有工具、库或模块能解决问题，就用它。重新发明轮子不仅浪费，还会引入新的 bug。Unix 工具箱经过数十年打磨，稳定、高效、可移植。但重用不等于盲目依赖——评估成熟度与维护性。

- **反面信号**：自己写 JSON 解析器、自己造日志框架、拒绝使用标准库

### #66 把软件工艺当作长期修炼

谦逊、好奇、诚实、沟通、创造力、坚持等个人特质，与技艺一同决定一个工程师的上限。持续学习与反思是工艺精神的核心。工程文化（简洁、透明、协作、务实）比任何具体技术都更持久。

- **反面信号**：停止学习新范式、拒绝 code review、把"能跑"当作完成标准

## 附录：原则间的核心张力

应用原则时注意以下张力，**回到原则 #1（管理复杂度）做最终裁决**：

1. **简单性 (#4) vs 防御式编程 (主题六)**：过度防御增加复杂度
2. **信息隐藏 (#9) vs 透明性 (#7)**：隐藏实现 vs 可观察内部
3. **先正确再高效 (#8) vs 性能要求 (#63)**：有时需为性能妥协
4. **避免过早优化 (#64) vs 架构期考虑性能**：架构决策难事后改
5. **组合优于继承 (#16) vs 既有代码库**：旧代码可能重度依赖继承
6. **silence is golden (#52) vs 可观察性 (#7)**：静默成功 vs 日志详尽
