# Coding Art — 编程工艺原则集

## Overview

编程工艺核心原则集——融合《代码大全 2》(Steve McConnell) 与《UNIX 编程艺术》(Eric S. Raymond) 两部经典，提炼出 66 条通用编程原则与范式，覆盖从心智态度到具体编码、从设计到测试、从性能到工程文化的全流程。

这些原则不是教条，而是经过数十年工业实践检验的"启发式方法"（heuristics）。它们之间存在张力——例如"简单优先"与"防御式编程"在某些场景需要权衡。把它们当作决策时优先参考的清单，而不是机械执行的规则。

## When to Use

在以下场景主动应用本原则集：

- **写代码时**：作为编码前的对照清单，确保不违反核心原则
- **代码评审（Code Review）时**：作为评审维度框架
- **设计与架构决策时**：在松耦合/强内聚/复杂度/简单性之间权衡
- **重构时**：识别违反原则的代码并改造
- **调试与优化时**：遵循"测量再优化""科学调试"等原则
- **回答编程最佳实践问题时**：作为原则性参考与评审框架

## How to Use

### 1. 框架思维：10 大主题

66 条原则按 10 大主题组织，每个主题对应软件工艺的一个维度：

| 主题             | 关注点                     | 原则数 |
| ---------------- | -------------------------- | ------ |
| 一、心智与哲学   | 编程的根本态度与价值观     | 8      |
| 二、设计与模块化 | 架构、模块、抽象、耦合     | 10     |
| 三、命名与表达   | 代码可读性的第一道关       | 8      |
| 四、函数设计     | 子程序级别的工艺           | 8      |
| 五、控制流与数据 | 语句级别的正确性           | 6      |
| 六、防御式编程   | 错误处理与边界             | 6      |
| 七、接口与协议   | 对外契约与协作             | 6      |
| 八、并发与状态   | 状态管理与并发模型         | 4      |
| 九、测试与质量   | 质量保证与调试方法论       | 6      |
| 十、优化与工程   | 性能、工具、文档与工艺修炼 | 4      |

### 2. 详细原则清单

完整的 66 条原则（含每条的解释、反例、出处）存放在 `references/principles.md`。

**触发时机**：

- 当任务涉及编码、设计、评审、重构时，**主动加载** `references/principles.md` 到上下文
- 当用户询问某类最佳实践（如"如何命名变量""怎么写好函数""如何调试"）时，加载并引用相应主题
- 当需要给出有依据的代码建议时，引用原则编号（如"依据原则 #12 SPOT"）

### 3. 应用流程

**写新代码时**：

1. 先对照"心智与哲学"（主题一）确认方向
2. 设计阶段应用"设计与模块化"（主题二）
3. 编码阶段依次应用主题三~六
4. 完成后用主题九做自检

**评审既有代码时**：

1. 先理解当前代码库的目标、约束、既有风格和测试方式
2. 识别违反"心智与哲学"的根本性问题
3. 逐层下沉到模块、函数、语句级别
4. 检查测试覆盖、错误处理、可观测性和性能证据
5. 按严重程度输出问题，并给出文件/位置、影响、对应原则和可执行建议

**回答"如何写好 X"类问题时**：

1. 在 `references/principles.md` 中检索相关主题
2. 引用 2~5 条最相关的原则
3. 结合具体代码示例说明

### 4. Agent 应用纪律

应用这些原则时，先服从当前代码库的真实约束，再使用原则做判断。不要机械展开 66 条清单。

- **尊重上下文**：优先遵守项目已有架构、语言惯例、团队规范、测试策略和用户明确目标。
- **选择性引用**：每次只选择与当前任务最相关的 3 到 7 条原则，避免把原则清单当成泛泛而谈的模板。
- **证据优先**：性能、安全、并发、可靠性类建议必须尽量基于 profiling、测试、复现路径、威胁模型或具体代码路径。
- **输出可执行**：代码评审应先列风险和影响，再给修改建议；不要只给抽象原则。
- **承认例外**：当原则之间冲突时，说明权衡理由。库、框架、语言版本和业务约束可以覆盖一般原则。
- **不替代官方文档**：涉及具体 API、框架行为、配置语义或版本迁移时，原则只能辅助判断，具体细节应以项目实现和当前官方文档为准。

### 5. 原则之间的张力

下列原则之间存在天然张力，应用时需结合上下文权衡，**而非机械套用**：

- **"简单优先"（#4）vs "防御式编程"（主题六）**：过度防御会增加复杂度
- **"信息隐藏"（#9）vs "透明性"（#7）**：隐藏实现 vs 可观察内部
- **"先正确再高效"（#8）vs 性能要求**：有时需为性能做妥协
- **"避免过早优化"（#64）vs "架构期考虑性能"**：架构决策难以事后修改
- **"组合优于继承"（#16）vs 现实代码库**：旧代码可能重度依赖继承

遇到张力时，回到原则 #1（管理复杂度是首要使命）做最终裁决。

## Source Attribution

本 skill 的 66 条原则提炼自以下两部经典：

- **《代码大全 2》(Code Complete 2)** — Steve McConnell 著，软件构建工艺的百科全书
- **《UNIX 编程艺术》(The Art of UNIX Programming)** — Eric S. Raymond 著，Unix 哲学与设计传统

部分原则融合了两书的共同智慧（如"先正确再高效"在两书中均有论述），部分原则为单书特色（如 SPOT、文本接口、silence is golden 来自 Unix 编程艺术，防御式编程、代码检查、科学调试来自代码大全）。具体出处见 `references/principles.md` 每条原则的标注。
