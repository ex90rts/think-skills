# Methodology: System Thinking

System thinking views problems as parts of interconnected systems rather than isolated points. Many issues are not caused by one object or one event; they emerge from relationships, feedback, nested systems, and delays.

## Single-Point Thinking Versus System Thinking

Single-point thinking sees many things as separate points. It asks: "What is the thing causing this?"

System thinking asks:

- What system is producing this result?
- What variables are changing?
- How do causes connect across time?
- What feedback loops amplify or stabilize behavior?
- What delays hide the real effect?
- Which stakeholders create or absorb consequences?

## Five Elements Of A System

### 1. Variables

Variables are elements that change over time. They can be quantitative or qualitative, but they should be observable enough to reason about.

Good variables:

- Decision latency.
- Customer trust.
- Team fatigue.
- Product reliability.
- Inventory pressure.
- Cash runway.
- Compliance burden.

Weak variables:

- "Bad culture"
- "Management problem"
- "User issue"

Convert weak labels into observable variables.

### 2. Causal Chains

Causal chains connect variables:

`A increases -> B decreases -> C increases`

Every causal link should include a mechanism. If the mechanism is unclear, mark it as uncertain rather than pretending the link is known.

### 3. Reinforcing Loops

Reinforcing loops amplify change. They can create virtuous cycles or vicious cycles.

Examples:

- Better product -> more users -> more data -> better product.
- More workload -> more mistakes -> more rework -> more workload.

Ask whether the loop drives growth, decline, escalation, or lock-in.

### 4. Balancing Loops

Balancing loops resist change, cap growth, or pull the system back toward a target.

Examples:

- More growth -> more operational strain -> slower growth.
- More errors -> stricter review -> fewer errors.
- Higher price -> lower demand -> pressure to lower price.

Ask what limit, correction, or resistance is being created.

### 5. Delay Effects

Delays are time gaps between cause and result, action and feedback, problem and signal, or cost and consequence.

Common delays:

- A training program improves performance only after weeks.
- Technical debt appears cheap now but slows future work.
- Customer dissatisfaction appears before churn metrics move.
- Hiring solves capacity later, but onboarding increases workload first.

Delays often cause overcorrection, premature abandonment, or false confidence.

## Concentric View

When judging a plan or issue, examine nested layers:

1. Problem or issue.
2. Causes.
3. Results.
4. Entities or stakeholders.

This prevents missing important elements. A plan that solves the immediate issue but ignores stakeholders or delayed results may fail inside the larger system.

## Intervention Principles

Prefer interventions that change system behavior rather than only treating symptoms.

Potential leverage points:

- Change information flow.
- Change incentives.
- Change feedback speed.
- Change decision rights.
- Change constraints.
- Add a balancing loop.
- Weaken a harmful reinforcing loop.
- Strengthen a beneficial reinforcing loop.
- Reduce harmful delays.

## Quality Bar

A strong system analysis:

- Defines a boundary.
- Names stakeholders.
- Converts vague issues into variables.
- Shows causal chains with mechanisms.
- Identifies at least one reinforcing loop and one balancing loop when present.
- Checks delay effects.
- Separates known links from uncertain links.
- Produces leverage points and watch indicators.

A weak system analysis:

- Lists factors without relationships.
- Finds only one root cause.
- Ignores stakeholders.
- Ignores time delays.
- Recommends a symptom fix without checking feedback effects.
- Treats all links as equally certain.
